교통 관리를 지원하기 위해 플릿 제어는 Order로 전달된 경로를 두 부분으로 나눌 수 있습니다.

- *"Base"*: 모바일 로봇이 주행할 수 있도록 정의된 경로입니다. base 경로의 모든 노드와 edge는 이미 플릿 제어가 모바일 로봇에 대해 released했습니다. base의 마지막 노드를 **decision point**라고 합니다.
- *"Horizon"*: decision point 이후 모바일 로봇이 주행할 것으로 플릿 제어가 현재 계획한 경로입니다. horizon 경로는 아직 플릿 제어에 의해 released되지 않았습니다.

base에 더 이상 노드와 edge가 추가되지 않으면 모바일 로봇은 decision point에서 정지해야 합니다. 원활한 이동을 위해 교통 상황이 허용하면 모바일 로봇이 decision point에 도달하기 **전에** 플릿 제어가 base를 연장하는 것이 좋습니다.

MQTT는 비동기 프로토콜이고 무선 전송은 신뢰할 수 없으므로 **base는 변경할 수 없습니다.** 따라서 플릿 제어는 base가 모바일 로봇에 의해 이미 실행되었다고 가정해야 합니다. 이후 절에서 Order 취소 절차를 설명하지만, 위와 같은 통신 한계로 이 역시 신뢰할 수 없는 것으로 간주됩니다.

플릿 제어는 변경된 노드·edge 목록을 포함한 갱신된 경로를 모바일 로봇에 보내 horizon을 변경할 수 있습니다. horizon 경로 변경 절차는 그림 3에 나와 있습니다.

또한 노드 g만 base 노드로 다시 보내집니다.
base는 변경할 수 없으므로 노드 f와 d를 재전송하는 것은 유효하지 않습니다.

모바일 로봇에 현재 사용 가능한 정보를 보여 줍니다.

decision point(그림 6의 노드 g)의 내용이 바뀌지 않는 것이 중요합니다. 즉 동작·편차 범위 등을 다시 보내야 합니다(그림 7, `orderUpdateId` 1 참고).

![Figure 8 The process of accepting an order or orderUpdate](https://raw.githubusercontent.com/VDA5050/VDA5050/main/assets/process_order_update.png)
>그림 8 — Order 또는 order update 수락 절차.

1) **수신 Order가 유효한가?**:
형식과 JSON 데이터 형식이 모두 올바른가?

2) **수신 Order가 새 Order인가 현재 Order의 갱신인가?**:
수신 Order의 `orderId`가 모바일 로봇이 현재 보유한 Order의 `orderId`와 다른가?

3) **모바일 로봇이 유휴이고 갱신을 기다리지 않는가?**:
모바일 로봇이 [6.6.8 Idle state of the mobile robot](/spec/6/6.6#668-idle-state-of-the-mobile-robot)에 따른 유휴 상태이고 갱신을 기다리지 않는가? State 안에 Order horizon의 노드·edge 및 해당 동작 상태도 포함되므로, 모바일 로봇은 여전히 horizon을 가지고 갱신을 기다리며 Order를 실행 중일 수 있다.

4) **OrderUpdateId가 0인가?**: 새 Order의 `orderUpdateId`가 0인가?

5) **새 Order의 시작이 현재 위치에 충분히 가까운가?**: 모바일 로봇이 이미 노드 위에 서 있거나 노드의 편차 범위 안에 있는가([6.1.1 Concept and logic](#611-concept-and-logic))?

6) **수신 Order 갱신이 구식인가?**: `orderUpdateId`가 모바일 로봇에 있는 것보다 작거나 같은가?

7) **cancelOrder 뒤의 Order 갱신인가?**: 취소된 Order에 대한 Order 갱신은 플릿 제어가 더 보내지 않아야 하고 모바일 로봇이 수락해서도 안 된다.

8) **수신 Order 갱신이 모바일 로봇에 있는 것과 같은가?**: `orderUpdateId`가 모바일 로봇에 있는 것과 같은가?

9) **수신 갱신이 현재 실행 중인 Order의 유효한 연속인가?**: 수신 Order의 첫 노드가 Order 갱신 장에 따른 현재 decision point인가? 모바일 로봇은 여전히 이동 중이거나 이전 Order 갱신에서 released된 base와 관련된 동작을 실행 중이거나 horizon을 가지고 Order 연속을 기다린다. 이 경우 Order 갱신은 새 base의 첫 노드가 이전 base의 마지막 노드와 같을 때만 수락된다.

10) **수신 갱신이 이전에 완료된 Order의 유효한 연속인가?**: 수신 Order의 첫 노드가 Order 갱신 장에 따른 현재 decision point인가? 모바일 로봇은 더 이상 동작을 실행하지 않고 Order 연속을 기다리지도 않는다(관련 동작을 모두 포함한 base를 완료했고 horizon이 없음). 이 경우 Order 갱신은 새 base의 첫 노드가 이전 base의 마지막 노드와 같을 때만 수락된다.

11) **채우기/추가**: `actionStates`/`nodeStates`/`edgeStates`에 새 상태를 채우거나 덧붙인다.

#### 6.1.2.1 Finishing an order  

모바일 로봇이 Order의 마지막 노드를 통과하고 Order 관련 이동과 동작을 모두 마치면 유휴 상태이며 새 Order를 받을 준비가 되어 있어야 합니다([6.6.8 Idle state of the mobile robot](/spec/6/6.6#668-idle-state-of-the-mobile-robot) 참고).
